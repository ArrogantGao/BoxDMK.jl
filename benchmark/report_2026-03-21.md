# BoxDMK.jl vs Fortran Benchmark Report

Date: 2026-03-21

## Executive Summary

The Julia solver has a **pre-existing correctness bug** that produces wrong results for volume potentials. The bug exists in the original code before any optimizations and manifests as non-convergent target potentials when eps is refined. The local (near-field) computation produces correct values matching Fortran, but the proxy/PW pathway produces values 8 orders of magnitude too large that don't properly cancel.

The recommended path forward is to use the Fortran compiled library (`libboxdmk.so`) via `ccall` for both tree building and solving, while keeping the Julia wrapper for API ergonomics.

## Benchmark Configuration

Problem: Laplace kernel, 3D, norder=16, eps=1e-6, boxlen=1.18, eta=0.0
Source: Sum of 2 Gaussians (Laplacian of Gaussians), matching Fortran bench_fortran_same_problem.f90

### Self-Convergence Test (pre-existing bug)

```
eps=1e-2: target_pot = [3.73e-02, 5.48e-02, -3.35e-01]
eps=1e-3: target_pot = [3.15e+01, 3.45e+01,  3.64e+00]
eps=1e-4: target_pot = [1.38e-01, 6.57e-02,  6.29e+01]
```

Values jump wildly — no convergence. This bug exists in the ORIGINAL code before optimizations.

## Root Cause Analysis

1. **Correctness bug (pre-existing)**: The proxy/PW pathway does not produce correct volume potentials. The PW conversion tables `tab_p2p @ tab_c2p` have condition number ~50-60 and the round-trip amplifies by ~25,000x. This is expected behavior (PW is overcomplete), but the kernel FT attenuation and the local correction must exactly cancel the amplification. Something in this cancellation is broken.

2. **Tree difference**: Julia produces 153 boxes vs Fortran's 1129 for the same problem. The Julia tree builder uses a different refinement algorithm, different error metric, and `_MAX_TREE_LEVELS=6`. This affects both accuracy and timing.

3. **Performance gap**: Even accounting for correctness issues, the Julia code is 258x slower than Fortran, dominated by:
   - Local/near-field: 587x slower (120s vs 205ms) — millions of tiny sparse tensor products
   - PW conversions: 100-341x slower — tensor products with porder=30 ↔ npw=44
   - Upward/downward passes: 65-95x slower — tensor products with porder=30

## Recommended Next Steps

### Phase 1: Use Fortran library via ccall (immediate)

Create a Julia wrapper that calls the Fortran C API:
- `boxdmk_vol_tree_mem` + `boxdmk_vol_tree_build` for tree construction
- `boxdmk_bdmk` for the solver

This gives correct results + Fortran speed immediately while keeping Julia API ergonomics.

### Phase 2: Debug Julia solver correctness (separate effort)

The pre-existing bug needs investigation:
- The PW pathway's charge→PW→kernel_ft→PW→proxy→potential chain must be verified step by step
- Compare intermediate values (proxy_charges, PW expansions, kernel_ft) against Fortran at each step
- The issue may be in the kernel FT computation, the SOG decomposition, or the PW conversion table construction

### Phase 3: Optimize Julia pure-Julia solver (after correctness is fixed)

Only relevant after the correctness bug is fixed. Current bottlenecks:
- Local/near-field (120s): needs algorithmic improvement or Fortran ccall
- PW conversions (25s): need Fortran-style data layout (no transposes)
- Upward/downward (7s): need Fortran-style data layout
