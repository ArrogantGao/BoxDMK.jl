## Executive Summary

The current Julia benchmarking scripts do not perform a like-for-like comparison against the Fortran benchmark. They use a different source function and a different tree-refinement parameter than the Fortran executable, so the reported Julia-vs-Fortran numbers are not trustworthy as written.

On this machine, the native Fortran benchmark finished quickly:

- tree build: `0.251 s`
- solve: `0.525 s`
- total: `0.776 s`
- boxes: `1129`
- levels: `6`

The Julia translation did not finish the same benchmark configuration within `30 s` after warm-up. The timed-out stack was inside [`src/planewave.jl:167`](/mnt/home/xgao1/codes/BoxDMK.jl/src/planewave.jl#L167) through [`src/planewave.jl:228`](/mnt/home/xgao1/codes/BoxDMK.jl/src/planewave.jl#L228), called from [`src/solver.jl:179`](/mnt/home/xgao1/codes/BoxDMK.jl/src/solver.jl#L179). That gives a conservative lower bound of `> 38.7x` slower than the Fortran total runtime for the exact benchmark case.

## Review Findings

1. High: the Julia benchmark scripts claim to match the Fortran benchmark, but the RHS does not match the Fortran executable.
   - Julia benchmark RHS uses Gaussian terms like `exp(-r^2 / s^2) / (π s)` in [`benchmark/run_benchmarks.jl:16`](/mnt/home/xgao1/codes/BoxDMK.jl/benchmark/run_benchmarks.jl#L16), [`benchmark/quick_benchmark.jl:19`](/mnt/home/xgao1/codes/BoxDMK.jl/benchmark/quick_benchmark.jl#L19), and [`benchmark/benchmark.jl:12`](/mnt/home/xgao1/codes/BoxDMK.jl/benchmark/benchmark.jl#L12).
   - The Fortran benchmark callback uses `exp(-rr / sigma) * (-2*ndim + 4*rr/sigma) / sigma` in [`bench_fortran_same_problem.f90:100`](/mnt/home/xgao1/codes/boxdmk/benchmark/bench_fortran_same_problem.f90#L100) through [`bench_fortran_same_problem.f90:128`](/mnt/home/xgao1/codes/boxdmk/benchmark/bench_fortran_same_problem.f90#L128).
   - Impact: the current Julia-vs-Fortran timing table is comparing different problems.

2. High: the Julia benchmark helpers also miss the Fortran tree-refinement setting.
   - The Julia tree builder defaults `eta = 1.0` in [`src/tree.jl:232`](/mnt/home/xgao1/codes/BoxDMK.jl/src/tree.jl#L232).
   - The benchmark helper in [`benchmark/run_benchmarks.jl:77`](/mnt/home/xgao1/codes/BoxDMK.jl/benchmark/run_benchmarks.jl#L77) does not pass `eta`, so it inherits `1.0`.
   - The Fortran executable explicitly sets `eta = 0.0d0` in [`bench_fortran_same_problem.f90:34`](/mnt/home/xgao1/codes/boxdmk/benchmark/bench_fortran_same_problem.f90#L34).
   - Impact: even with the same RHS, the adaptive trees would still differ.

3. Medium: the project’s “cross-validation” test is not a Fortran cross-validation test.
   - [`test/test_cross_validation.jl:25`](/mnt/home/xgao1/codes/BoxDMK.jl/test/test_cross_validation.jl#L25) only checks output shape, finiteness, and loose self-consistency.
   - It never calls the Fortran library or executable.
   - Impact: CI currently has no automated guard against Julia/Fortran divergence.

4. High: the exact Fortran-sized Julia benchmark stalls in plane-wave setup.
   - The 30-second timeout terminated during `build_pw_shift_matrices` and `setup_planewave_data` in [`src/planewave.jl:167`](/mnt/home/xgao1/codes/BoxDMK.jl/src/planewave.jl#L167) through [`src/planewave.jl:228`](/mnt/home/xgao1/codes/BoxDMK.jl/src/planewave.jl#L228), called from [`src/solver.jl:179`](/mnt/home/xgao1/codes/BoxDMK.jl/src/solver.jl#L179).
   - Impact: the current implementation is not competitive on the reference case, and the hotspot is localized enough to profile directly.
